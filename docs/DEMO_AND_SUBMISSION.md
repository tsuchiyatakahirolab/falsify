# Demo and Submission Plan

## Three-minute narrative

### 0:00–0:20 — Problem
A report can look credible, contain real citations, and still make claims that the cited evidence does not establish.

### 0:20–0:45 — Product
Falsify does not "review the paper." It breaks the argument into claims and asks what would make each claim fail.

### 0:45–1:45 — Live workflow
1. Submit a public statement or document.
2. Show atomic claims.
3. Open one Claim Card.
4. Show "What would make this wrong?"
5. Show supporting evidence.
6. Show contradictory/qualifying evidence.
7. Show source/citation or inference issue.
8. Show qualified verdict.

### 1:45–2:20 — Flagship narrative
Use a real public China-to-Japan strategic narrative:
- acknowledge a supported factual core;
- expose a separate historical analogy or causal/inferential leap;
- show why the distinction matters.

### 2:20–2:40 — Challenge Falsify
Trigger "Challenge this finding."
Show that the system can qualify or defend its own conclusion based on evidence.

### 2:40–3:00 — Build Week requirement
Voiceover must explicitly state:
- what was built;
- how Codex was used;
- how GPT-5.6 is used in the product.

## Submission checklist

Official Build Week guidance requires:
- working project;
- category;
- project description;
- public YouTube demo under 3 minutes with audio covering Codex and GPT-5.6 use;
- code repository;
- public repo has a relevant open-source license;
- README includes setup instructions and sample data if needed;
- README highlights where Codex accelerated work, key decisions, and GPT-5.6/Codex usage;
- `/feedback` Codex Session ID for the session where the majority of core functionality was built;
- for developer tools/plugins, installation/platform/testing instructions.
